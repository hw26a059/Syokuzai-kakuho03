# 食材目利きゲーム 仕様書：仕様決定ログ及びロードマップ

すべての検討事項について合意がなされ、仕様が完全に確定いたしました。以下にその決定ログと、次の開発フェーズに向けたロードマップをまとめます。

## 1. 仕様決定ログ

### クイズの出題方法
- **出題形式**: 3枚の写真から正しい状態のものを選ぶ「3択写真形式」を最優先とします。
- **写真の条件**:
  - スーパーなどで実際に売られている「パックやトレイ、袋に入った状態」のものを写真として使用。
  - 選択肢は「極めて新鮮・旬のもの（正解）」「少し時間が経ってドリップ等が出ているもの（不正解）」「旬を過ぎたり一部変色や傷みが見られるもの（不正解）」の3枚で構成します。
- **出題順**: プレイヤーがカスタマイズした食材の並び（または定義順）に基づき、肉・魚・野菜といったカテゴリを跨いで連続で出題します。
- **制限時間**: 1問あたり1分（60秒）とし、時間に追われすぎずじっくり考えられる設計とします。

### 料理システムと図鑑の挙動
- **食材のカスタマイズ**: ベースとなる料理レシピを選択後、チェックボックスにより具材の変更・追加を自由に行えます（ただし相性が著しく悪い食材は選択肢から除外されます）。
- **美味しさの判定**: クイズの正解率が50%以上であれば美味しい料理が完成。50%未満の場合のみ「不味く（失敗扱い）」なります。
- **自由命名と図鑑登録**: 美味しく完成した料理に対して、プレイヤーが自由に料理名（例：シーフードカレー）を命名できます。
- **図鑑の分類**: プレイヤーが命名した料理は、ベースとなる「レシピ（タグ）」（例：「カレーライス」「肉じゃが」等）ごとにグループ分けされて図鑑にコレクションされます。失敗した不味い料理は図鑑に登録されません。

### スコア・ランキングシステム
- **ユーザー識別**: プレイヤーは初回起動時に「ニックネーム」を入力・登録（簡易ログイン）します。
- **ランキング基準**: ニックネームに紐づく「累計スコア」をサーバーで管理し、累計獲得スコアの高さで競い合います。
- **ランキング規模**: サーバー上に蓄積されたデータのうち、**上位50名**をランキングに表示します。

## 2. 次のフェーズ：開発ロードマップ

完成した仕様書（`/spec/` 配下のマークダウン群）を、開発用のAIモデルまたはエンジニアに受け渡すことで、ゲーム制作へ進めることが可能です。

1. **第1段階：フロントエンド（画面構成・UI）の実装**
   - ニックネーム登録画面、レシピ選択＆食材カスタマイズ画面、クイズ＆解説画面、料理結果画面、図鑑画面、ランキング画面。
2. **第2段階：データ（クイズ・写真・解説）の流し込み**
   - 16以上のレシピ候補（カレーライス、肉じゃが、八宝菜など）に必要なクイズデータ、3択の画像アセット、ミニ解説テキスト。
3. **第3段階：バックエンド（ランキングサーバー・データベース）の構築**
   - ニックネームごとの累計スコアを永続化するデータベース（Firestore等）と、ランキングを上位から順に取得するAPI。
4. **第4段階：演出の強化・テスト**
   - 料理が不味くなってしまったときのビジュアルや、美味しい料理ができて図鑑に登録される際のエフェクト（motion等）追加。
